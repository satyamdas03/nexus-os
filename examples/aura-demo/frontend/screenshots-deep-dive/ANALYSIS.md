# AURA Phase 2 — Hands-on Feature Analysis (2026-06-20)

Ran the backend + frontend locally, exercised every major feature through the browser and direct API calls, and captured 27+ screenshots.

**Servers tested**
- Backend: `uvicorn main:app` on `http://127.0.0.1:8000`
- Frontend: `npm run dev` on `http://localhost:3000`
- Book: 34,000 synthetic portfolios freshly generated by `ensure_book()`

## 1. Command Centre (`/`)

**Screenshots**: `01-command-centre-initial.png`, `03-command-centre-filter-adviser.png`, `04-command-centre-search.png`, `06-command-centre-after-tick.png`, `21-after-admin-reset.png`

### What works
- Page loads and renders the full 34k book summary: `$119.32B` AUM, `80.0%` green, `5,100` attention, `1,700` breached, `8,013` breaches.
- Heatmap (Recharts Treemap) renders green/orange/red blocks sized by FUM.
- Urgent Triage Log lists red/orange portfolios on the right.
- Market panel shows day, price sparkline, and `STOP CLOCK` / `AUTO-ADVANCE` / `TICK` controls.
- Clicking `TICK` advances the market day and updates the summary.

### What needs attention
- The status filter is a `<select>` with options `Breached`, `Attention`, `Needs Action` (not a chip), so a naive "click BREACH chip" automation misses it. The select itself works when used.
- The adviser filter can be very restrictive: selecting one adviser can collapse the heatmap to only a handful of portfolios even though the book has 34k. This is technically correct (only that adviser's clients), but the UX can look broken because the first page is nearly empty.
- After admin reset, the client-side summary refetches, but the market panel label briefly shows the old day until the client fetch completes (minor stale flash).

## 3. Workbench (`/portfolio/{id}/workbench`) — ✅ FIXED

**Screenshots**: `workbench-red-initial.png`, `workbench-proposed.png`, `workbench-approved-green.png`, `workbench-approved-lower.png`

### What works
- Page loads portfolio state and displays the assurance checklist.
- `Propose a fix` now generates a real, breach-aware trade list for red portfolios.
- The live verify/approve flow records trades in the shadow state and flips the portfolio green.
- The post-approval banner shows the rules-engine re-check result (status GREEN).
- Audit trail appends `remediate_propose`, `verify`, and `approve` events.

### Fix applied
- Rewrote `agents/llm.py` `MockLLM._mock_remediate_trades()`:
  - Added `_extract_first_json_object()` that scans brace balance and ignores trailing text, so retry prompts (facts JSON + breaches JSON concatenated) parse correctly.
  - Computes required sell market value per offending holding from each breach instead of blindly selling 60%.
  - Handles `approved_universe` / `esg_exclusions` breaches by selling 100% of offending holdings.
  - Handles `min_cash` by holding proceeds as cash.
  - Handles `min_liquid_pct` by buying approved tier-1 tickers (split across several names to avoid creating a new concentration breach).
  - Redeploys remaining proceeds into underweight approved asset classes.
- All 138 backend tests pass.
- Resolution rate on day-0 red portfolios: **60/102 (58.8%)** resolve green after one remediate call (20 of those via the single retry). `c00011` resolves on the first attempt and is the cleanest demo portfolio.

### Verified demo path (c00011)
1. `/portfolio/c00011/workbench` shows status **BREACH** (Single holding 30% > 20% cap, min_cash, US region).
2. `Propose a fix` returns 7 sells (VTI, BTC, HYG, IEF, XLF, XLV, XLY) totalling ~$1.1M; Assurance Check shows **0 BREACH / 32 COMPLIANT**.
3. `Approve & Log` records the intent; the banner reads **APPROVED + LOGGED — status GREEN**.
4. `POST /portfolio/c00011/check` confirms **green** with 0 breaches.

### What still needs attention
- Explain button still returns MockLLM placeholder text (same as Diagnosis).

## 4. Hermes Mission Control (`/hermes`) — ✅ MOSTLY WORKS

**Screenshots**: `16-hermes-initial.png`, `17-hermes-scanning.png`, `17-hermes-scanning-0.png`, `19-hermes-strategy.png`, `20-hermes-history.png`, `22-hermes-queue-ready.png`, `23-hermes-row-expanded.png`, `26-hermes-bulk-ready.png`, `27-hermes-bulk-approved.png`

### What works
- **Full-book scan**: POST `/hermes/scan` processes all 34,000 portfolios asynchronously. On a clean day-0 book it remediated 90 portfolios in ~4 seconds.
- **Queue rendering**: ranked list shows `prior_status` → `post_status` (red → green), FUM, trade count, rationale, and confidence.
- **Row expansion** fetches live verification via `/portfolio/{id}/verify` and renders the per-rule checklist with `VerifyPanel`.
- **Bulk approve** (`Approve all verified`) records trades in the shadow state and flips portfolios green. Verified for `c12253`, `c32947`, `c14021` — all became green and stayed green on reload.
- **Shadow state persistence is real**: approved trades live in the SQLite `state` table; reloading `/portfolio/{id}` after approval returns the green effective portfolio.
- **Strategy panel** displays `strategy.yaml` v1 variables and rationales.
- **Assurance cage diagram** is rendered with the two-tier cage explanation.

### What needs attention
- **Bulk approve UX inconsistency**: the frontend removes all items from the local queue after a successful bulk call (`setItems([])`), but the backend may still have more than 50 queued items. The UI shows "0 assets in queue" while a refresh reveals more. The bulk call should probably refetch the queue or only clear the approved slice.
- **Individual Approve button**: the component code is correct and sends `q.trades` to `api.approve`; automated selection was tricky because the bulk button also contains "Approve" text.
- **Hermes does not fix every portfolio**: ~90 remediated / ~1,481 missed on day 0. This is acceptable for a v1 strategy (the whole point is reflection/self-improvement), but the demo script should pick a portfolio that Hermes can fix.

## 5. Admin Reset & Market Simulation

### What works
- `POST /admin/reset` clears `state`, `hermes_queue`, `scan_jobs`, `drift_events`, `status_history`, rewinds clock to day 0, and recomputes day-0 status.
- `POST /market/tick` advances the clock and re-runs the drift monitor; status counts change as prices move.
- Market panel controls are wired and visible.

### What needs attention
- After reset, the Command Centre shows the correct summary once the client refetch completes, but the market-day label can flash stale for a moment.
